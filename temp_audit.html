1.3 Request Lifecycle — Leave Submission

The following trace covers the full path from browser to database and back, for POST /api/v1/leaves:

  1. HTTPS POST arrives at /api/v1/leaves → received by index.js, forwarded to leaveRoutes.js
  2. verifyToken middleware fires: decodes JWT, checks signature and expiry, queries DB to confirm user record is active
  3. Zod validation middleware fires: validates request body shape and types before controller is reached
  4. leaveController.submitLeave executes: business logic runs, new leave_request row inserted into SQLite
  5. HTTP 201 response returned to client
  6. Independently, node-cron in automationService.js scans every minute for pending leave_requests where submitted_at + 30 minutes ≤ NOW, and auto-approves matching rows via a conditional UPDATE


Section 2 — Security Findings

The authentication foundations (JWT validation, role enforcement, DB-verified user checks, startup secret guard) are correctly implemented. Three issues remain.

HIGH

File: client/src/context/AuthContext.js

Line(s): Inferred — localStorage.setItem call

PROBLEM

JWTs are stored in localStorage on the frontend. Any JavaScript running on the same origin — including injected scripts from XSS attacks — can read localStorage directly. This is a well-documented, widely exploited vulnerability pattern.

IMPACT

An attacker exploiting any XSS entry point (e.g., a stored malicious announcement, an unsanitised input field) could call localStorage.getItem("token"), extract the JWT, and replay it from a different machine to impersonate an Admin or HOD indefinitely until token expiry.

FIX

Move token storage to an httpOnly, Secure, SameSite=Strict cookie set by the backend on login. Remove all localStorage.setItem / getItem token calls from the frontend. The frontend never touches the token directly — the browser sends the cookie automatically on every request.

// server/controllers/authController.js — replace json token return with cookie

res.cookie('token', token, {

httpOnly: true, // JS cannot read this

secure: process.env.NODE_ENV === 'production', // HTTPS only in prod

sameSite: 'strict', // CSRF protection

maxAge: 7 * 24 * 60 * 60 * 1000 // 7 days in ms

});

res.json({ user: { id, name, role } }); // no token in body

MEDIUM

File: server/routes/leaveRoutes.js (and other resource routes)

PROBLEM

Input validation via Zod middleware is applied to /login but has not been extended to all POST and PUT routes. Controllers are performing manual, ad-hoc validation internally — which is inconsistent and easy to miss on new routes.

IMPACT

Malformed or adversarial payloads (oversized strings, wrong data types, unexpected fields) can reach controller logic and either crash the application or produce corrupt database records. Without a consistent validation layer, coverage gaps are invisible until exploited.

FIX

Create a universal validate(schema) middleware wrapper and apply a Zod schema to every POST and PUT route definition. This catches invalid input at the boundary before any business logic runs.

// server/middleware/validate.js (extend to all routes)

const { z } = require("zod");

const validate = (schema) => (req, res, next) => {

const result = schema.safeParse(req.body);

if (!result.success) return res.status(400).json({ errors: result.error.flatten() });

req.body = result.data;

next();

};

// server/routes/leaveRoutes.js

const leaveSchema = z.object({

startDate: z.string().datetime(),

endDate: z.string().datetime(),

reason: z.string().min(10).max(500),

type: z.enum(["sick", "personal", "emergency"]),

});

router.post("/", verifyToken, validate(leaveSchema), submitLeave);

LOW

File: server/middleware/uploadMiddleware.js

Line(s): 28–35

PROBLEM

MIME type validation checks file.mimetype, which is the Content-Type header value supplied by the HTTP client — not the actual file content. A client can rename malware.exe to image.jpg, set Content-Type: image/jpeg, and bypass this check entirely.

IMPACT

Malicious executable files could be uploaded to the server. While they cannot be directly executed via a URL (uploads are not served as scripts), they may be exploitable via secondary attack vectors such as path traversal, symlink attacks, or if the upload directory is ever misconfigured to permit execution.

FIX

Read the first bytes of the file stream (magic bytes) using the file-type package to verify the actual file format independently of what the client claims.

npm install file-type

// server/middleware/uploadMiddleware.js

const { fileTypeFromBuffer } = require("file-type");

const validateMagicBytes = async (req, res, next) => {

const file = req.file;

if (!file) return next();

const type = await fileTypeFromBuffer(file.buffer);

const allowed = ["image/jpeg","image/png","application/pdf","application/vnd.openxmlformats"];

if (!type || !allowed.includes(type.mime)) {

return res.status(400).json({ error: "Invalid file type" });

}

next();

};

✅ The following previously-reported security issues are confirmed resolved:

  • JWT secret startup validation — index.js refuses to start in production if JWT_SECRET is weak or default
  • Rate limiting on auth endpoints — express-rate-limit is applied to /login and /forgot-password
  • verifyToken middleware queries the database on every request — stale/revoked tokens are rejected


Section 3 — Bug & Reliability Findings

MEDIUM

File: server/services/automationService.js

Line(s): 21–28

PROBLEM

The automation cron job correctly uses node-cron and database timestamp comparison to find expired leave requests. However, between the SELECT (finding pending leaves) and the UPDATE (marking them approved), there is no row-level atomic protection. Two simultaneous cron runs — possible in PM2 cluster mode — can read the same pending row and both attempt to process it.

IMPACT

In PM2 cluster mode with 2+ instances, the same leave request could be auto-approved twice, triggering duplicate substitute assignment workflows, duplicate notifications, and potential double-booking of a substitute teacher for the same lecture slot.

FIX

Add "AND status = 'pending'" to the UPDATE WHERE clause. SQLite's row-level write lock means only one UPDATE will match — the second will affect 0 rows, achieving atomic safety without external locking infrastructure.

// server/services/automationService.js

// Replace the two-step SELECT then UPDATE with a single conditional UPDATE:

await db.run(`

UPDATE leave_requests

SET status = 'auto_approved',

hod_decision_at = CURRENT_TIMESTAMP

WHERE id = ? AND status = 'pending'

`, [leave.id]);

// Then check changes to know if this process "won":

const result = await db.run(...);

if (result.changes === 0) return; // another process already handled this

LOW

File: server/config/db.js

Line(s): 1–14

PROBLEM

WAL mode is set via db.run("PRAGMA journal_mode = WAL;") but the sqlite3 library's db.run() is asynchronous and fire-and-forget by default. If the connection is used immediately after construction (before the PRAGMA callback fires), queries may run briefly in default journal mode.

IMPACT

Under normal startup this is unlikely to cause visible issues. However under heavy load at startup, or in test environments that immediately write to the database after connection creation, queries could transiently run without WAL mode active — causing unnecessary write locks.

FIX

Wrap the PRAGMA in a Promise and await it before exporting the database connection. This guarantees WAL mode is active before any other module can use the DB.

// server/config/db.js

const sqlite3 = require("sqlite3").verbose();

const initDb = () => new Promise((resolve, reject) => {

const db = new sqlite3.Database(process.env.DATABASE_URL);

db.run("PRAGMA journal_mode = WAL;", (err) => {

if (err) return reject(err);

db.run("PRAGMA synchronous = NORMAL;", (err2) => {

if (err2) return reject(err2);

resolve(db);

});

});

});

module.exports = { initDb };

// server/index.js

const { initDb } = require("./config/db");

const db = await initDb(); // guaranteed WAL before any route handler runs


Section 4 — Performance Findings

MEDIUM

File: server/services/automationService.js

Line(s): 43–45

PROBLEM

Auto-approval executes sequentially in a for...of loop: for (const leave of pendingLeaves) { await this.autoApproveLeave(leave); }. Inside autoApproveLeave, each call runs a SELECT and INSERT query for every affected lecture. These database calls are serialised — each one waits for the previous to complete.

IMPACT

If 50 leave requests expire at the same time (e.g., after a server outage and recovery), the cron job will block the Node.js event loop for the duration of 50 sequential database round-trips. Other incoming HTTP requests will queue behind this operation, causing visible latency spikes for all users.

FIX

Process auto-approvals inside a single database transaction and use Promise.all with a concurrency limit to parallelise independent leaves. This reduces wall-clock time from O(n) sequential to near-constant.

// server/services/automationService.js

const pLimit = require("p-limit"); // npm install p-limit

const limit = pLimit(5); // max 5 concurrent DB operations

const tasks = pendingLeaves.map(leave =>

limit(() => this.autoApproveLeave(leave))

);

await Promise.all(tasks);

LOW

File: client/src/App.jsx

Line(s): 6–53

PROBLEM

All 45 page components are statically imported at the top of App.jsx. Webpack/Vite bundles every imported module into the initial JavaScript payload. A student visiting the login page downloads code for AdminDashboard, PredictiveAnalytics, AuditLogs, and 41 other pages they will never see.

IMPACT

The initial JavaScript bundle is significantly larger than necessary. This increases Time to Interactive (TTI) — especially on mobile devices and slower connections. For a college environment where students may use shared WiFi or mobile data, this is a meaningful UX degradation.

FIX

Replace static imports with React.lazy() for all page components. Vite automatically code-splits lazy-imported modules. Wrap all routes in a single Suspense boundary with a loading spinner.

// client/src/App.jsx — replace static imports

import React, { lazy, Suspense } from "react";

// Before: import AdminDashboard from "./pages/AdminDashboard";

// After:

const AdminDashboard = lazy(() => import("./pages/AdminDashboard"));

const HodDashboard = lazy(() => import("./pages/HodDashboard"));

const StudentDirectory = lazy(() => import("./pages/StudentDirectory"));

// ... repeat for all 45 pages

// Wrap router outlet:

<Suspense fallback={<div className="loading-spinner" />}>

<Routes>

<Route path="/admin" element={<AdminDashboard />} />

{/* ... */}

</Routes>

</Suspense>


Section 5 — Architecture & Code Quality Findings

HIGH

File: server/controllers/lectureController.js

Line(s): Entire file (36 KB / ~900+ lines)

PROBLEM

This single file contains raw SQLite/Knex queries, conflict detection algorithms, timetable formatting logic, substitute matching rules, and HTTP response construction — all intermixed. This violates the Single Responsibility Principle and the separation of concerns that makes code testable and maintainable.

IMPACT

Regression risk is high: modifying conflict detection logic risks breaking timetable formatting. Unit testing any single behaviour requires loading the entire controller. New developers cannot understand where to make changes without reading hundreds of lines of mixed concerns.

FIX

Refactor into a four-layer architecture. This makes each layer independently testable and replaceable.

// Target architecture for lecture domain:

// 1. lectureRoutes.js — HTTP path definitions only

// 2. lectureController.js — req/res parsing, calls service, returns response (~50 lines)

// 3. lectureService.js — business logic: conflict detection, scheduling rules (~200 lines)

// 4. lectureRepository.js — all Knex/SQLite queries, no business logic (~150 lines)

// Example — lectureController.js after refactor:

const LectureService = require("../services/LectureService");

exports.getSchedule = async (req, res, next) => {

try {

const schedule = await LectureService.getScheduleForUser(req.user);

res.json(schedule);

} catch (err) { next(err); }

};

MEDIUM

File: client/src/pages/Dashboard.jsx (and other data-fetching pages)

PROBLEM

@tanstack/react-query is installed and the QueryClientProvider is mounted in main.jsx, but individual page components still use manual useState + useEffect + axios patterns for data fetching. This means the React Query infrastructure is present but not being used.

IMPACT

Manual fetch patterns duplicate loading state, error state, and refetch logic across 45 components. There is no automatic cache invalidation, background refetching, or request deduplication. Users see stale data unless they manually refresh. Race conditions between competing useEffect calls are common.

FIX

Migrate data-fetching pages from useEffect + useState to useQuery hooks. Start with the highest-traffic pages (dashboards, timetable) and work iteratively. The QueryClient cache handles deduplication and background refresh automatically.

// Before (manual pattern — duplicated across pages):

const [data, setData] = useState(null);

const [loading, setLoading] = useState(true);

useEffect(() => {

axios.get("/api/v1/lectures/my-timetable")

.then(r => { setData(r.data); setLoading(false); })

.catch(console.error);

}, []);

// After (React Query — caching, dedup, background refresh built in):

const { data, isLoading, error } = useQuery({

queryKey: ["timetable", user.id],

queryFn: () => axios.get("/api/v1/lectures/my-timetable").then(r => r.data),

staleTime: 30_000,

});

Section 7 — DevOps & Config Findings

MEDIUM

File: ecosystem.config.js

PROBLEM

PM2's ecosystem config may default to instances: "max" which spawns one Node.js process per CPU core. SQLite is designed for a single writer — while WAL mode allows concurrent reads, multiple processes writing simultaneously will cause SQLITE_BUSY errors that WAL mode does not fully resolve under cluster load.

IMPACT

Under cluster mode with heavy write traffic (bulk attendance marking, simultaneous leave submissions during end-of-semester periods), the application will produce intermittent 500 errors from database lock contention. These errors are difficult to reproduce in development but appear consistently in production.

FIX

Set instances: 1 in ecosystem.config.js while the database remains SQLite. If horizontal scaling is required, migrate to PostgreSQL — the knexfile.js Postgres config is already in place for this transition.

// ecosystem.config.js

module.exports = {

apps: [{

name: "lecture-manager",

script: "./server/index.js",

instances: 1, // REQUIRED while using SQLite

exec_mode: "fork", // not "cluster"

env_production: {

NODE_ENV: "production",

max_memory_restart: "512M"

}

}]

};
